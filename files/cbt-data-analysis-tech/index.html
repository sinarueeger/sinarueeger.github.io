Data Analysis Worrying - Part 2

May 3, 2021 3 min read

This blogpost is about solutions to tackle constant “data analysis worrying”.

Part 1 discussed a mental coping approach, while Part 2 will show more technical solutions and prevention techniques. Let’s dive right in!

1. Work organisation

Write code such that mistakes can be corrected and analysis rerun easily.

The reasoning: this allows you to be at ease and not afraid of mistakes. Mistakes will happen to the greatest data analysts, but if you have a code/computing environment set up where you can quickly make a change to a code snippet written 2 months ago, and rerun the analysis, then you won’t fear mistakes. It’s one thing less to worry about.

  • Use version control for code (and backup resulting data regularly and in a systematic way).

  • Use an R-package to help with the analysis project organisation: targets, ProjectTemplate, workflowr and probably lots more.

  • Apply best practices in scientific computing (contains valueable tips, among them data management, project organisation, software).

2. Breaks

Take breaks often, especially when debugging - a draining activity - or during deep work. This will help to keep the focus up - even though it seems like you are taking the focus away by leaving your desk briefly.

The important bit is, that you rest during the break. Pet your cat, drink a coffee, watch the cars pass.

The comic below by Julia Evans shows the value of taking breaks in the context of debugging, but seems legit during a normal work day too.

Debugging tips by Julia Evans.

Breathing

During the break, or when stressed, apply box breathing. When stressed, our breathing becomes shallow. I don’t know the science behind it, but any type of deep and regular breathing helps to calm down and refocus.

Pomodoro Technique

When to take breaks? The official pomodoro technique is: work 25 min, have a 5 min break.

There are tons of apps around for exactly this purpose, but I found my oven timer most effective (I really have to leave my desk to turn off the timer).

3. R packages for high confidence in data

Use modern tooling

When reading in and out tables, don’t rely on read.table. Instead, use a more modern tooling, such as data.table::fread or readr::read_table that have not only faster, but are better in recognizing column data types.

Automatic data sanity checking

Use R-package janitor to apply automatic data checks.

Use R-package skimr to summarize your data quickly.

5. Debugging

If you happen to run into bug follow these tips by Julia Evans. Above all - take a break!

Debugging & Attitude by Julia Evans.

Quick changes vs. systematical approach

Quick bug fixes do exist, but if it takes more than a few minutes, you may better be off to take a break and approach the bug more systematically later on.

I find walks in forests the best to think about bugs.

The power of debugging

Finally, debugging is not all bad. On the contrary, debugging might be an opportunity to refactor some code, rethink a method, learn something new (see love your bugs by Julia Evans) or simply becoming better at staying composed during debugging.
